# 链式代理开关

## 零、项目重写原则

1. **不依赖外部仓库**：所有规则在本项目内自行维护创建，不依赖第三方规则仓库。
2. **按组审查确认**：每一组规则先确认设计，再进入完整配置。
3. **最小可用原则**：只保留必要的规则，不盲目复制大量个人偏好列表。

---

## 一、设计思路

### 1.1 什么是链式代理

链式代理（也叫多级代理、跳板代理）是指：**流量先经过一个代理节点，再跳到另一个代理节点，最终到达目标**。

常见用途：
- 某些服务对住宅 IP 友好，先连机场再跳住宅节点
- 提高匿名性，多层代理跳转
- 绕过某些地区的访问限制

### 1.2 本项目的链式代理设计

**不做复杂的规则匹配，只做一个全局开关策略组**：

- 创建一个 `链式开关` 策略组
- 选项 1：`关闭`（即不走链式，直接走「通用选择」）
- 选项 2：`住宅节点 1`
- 选项 3：`住宅节点 2`
- ...（根据实际节点数量添加）

- 「通用选择」策略组的第一个选项改为 `链式开关`

这样：
- 默认链式开关选「关闭」→ 流量走普通节点
- 需要走链式时，把链式开关切到某个住宅节点 → 所有流量先经过住宅节点
- 不需要单独为每个服务写规则，一个全局开关搞定

## 二、具体策略组设计

### 2.1 链式开关策略组

```yaml
proxy-groups:
  # 全局链式开关
  - name: 链式开关
    type: select
    proxies:
      - 关闭        # 默认：不走链式
      - 住宅节点    # 只保留一个住宅IP选项

  # 通用选择（注意：第一个选项改成了链式开关）
  - name: 通用选择
    type: select
    proxies:
      - 链式开关    # ✅ 第一个选项是链式开关，默认走「关闭」
      - 新加坡-故转
      - 香港-故转
      - 日本-故转
      - 美国-故转
      - 其他-故转
      - 所有-自动
      - 所有-手动
      - 直连
      - 拒绝
```

### 2.2 重要设计原则：相同服务商必须在同一规则内 ✅

**避免同一个服务商的不同域名走不同节点，导致IP漂移**。

例如：
- 所有 Google 相关域名（google.com, youtube.com, gmail.com, drive.google.com 等）都在「开发服务」分组 → 走同一个节点
- 所有 AI 相关域名（openai.com, anthropic.com, gemini.google.com 等）都在「AI 服务」分组 → 走同一个节点
- 所有流媒体相关域名（netflix.com, disneyplus.com, spotify.com 等）都在「流媒体」分组 → 走同一个节点

**好处**：同一个服务商的所有请求始终从同一个IP出去，不会出现"看YouTube走新加坡IP，发Gmail突然走香港IP"的情况，降低风控风险。

### 2.3 使用方式

| 场景 | 操作 | 效果 |
|------|------|------|
| 正常使用 | 链式开关选「关闭」 | 流量走「通用选择」选中的普通节点 |
| 需要走住宅节点 | 链式开关选「住宅节点X」 | 流量先经过住宅节点，再出去 |

**不需要改任何规则**，只需要切换「链式开关」策略组的选择即可。

## 三、关于节点排除

### 3.1 普通节点池排除链式节点

所有 `include-all` 的普通节点池（如「所有-自动」、「新加坡-故转」等）**提前排除链式/住宅节点**，避免循环。

这点在第 1 组基础策略组中已经确认过。

### 3.2 链式节点单独管理

链式/住宅节点单独放在「链式开关」策略组中，不进入普通节点池。

---

## 四、注意事项

### 4.1 链式代理的性能影响

链式代理会增加延迟（多了一跳），建议：
- 只在需要时开启
- 选择延迟低的链式节点
- 测速确认速度可以接受再长期使用

### 4.2 链式节点配置

链式节点需要在 `proxies` 部分预先配置好，然后添加到「链式开关」策略组的选项中。

### 4.3 没有链式节点时

如果暂时没有链式节点，可以把「链式开关」策略组简化成只有「关闭」选项，后续有节点了再添加。
