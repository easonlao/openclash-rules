# 广告与拦截规则

## 零、项目重写原则

1. **不依赖外部仓库**：所有规则在本项目内自行维护创建，不依赖第三方规则仓库。
2. **按组审查确认**：每一组规则先确认设计，再进入完整配置。
3. **最小可用原则**：只保留必要的规则，不盲目复制大量个人偏好列表。

---

## 一、设计原则

### 1.1 最小可用，宁少勿滥

- 广告拦截不是本配置的核心目标（主要用浏览器插件）
- 只拦截**影响体验、耗流量、有隐私风险**的域名
- 不追求「全量拦截」，避免误杀正常网站
- 不使用第三方规则集（easylist 等），避免依赖和规则膨胀

### 1.2 分类管理

按优先级排序：
1. **Tracker / 隐私追踪** —— 最高优先级，必须拦
2. **视频/APP 内置广告** —— 次高优先级
3. **网页广告联盟** —— 普通优先级
4. **恶意域名 / 钓鱼** —— 有就加，没有就不加

## 三、规则位置

广告/拦截规则放在：
```yaml
rules:
  # 1. 私有网络（最高优先级）
  # 2. 回家域名（仅手机）
  # 3. 自用直连
  # 4. AI 服务规则（高优先级，避免封号）
  # 5. ✅ 广告/拦截规则 ← 在这里
  # 6. 业务分流
  # 7. 国内直连
  # 8. 兜底
```

**为什么在 AI 服务之后？**
- AI 服务分流错误后果更严重（封号），所以优先级更高
- 广告拦截即使漏了，最多是看到广告，不会影响使用

## 四、可扩展机制（不依赖外部仓库）

### 4.1 本地维护方式

- 广告域名统一放在本文档管理
- 发现新广告域名时，在这里补充后再更新配置
- 定期（如每月）review 一次规则，删除失效或误杀的

### 4.2 不使用 rule-provider 的原因

- 避免依赖外部仓库的可用性和更新频率
- 避免规则集过大影响匹配性能
- 自行维护的少量规则更可控，更了解每条规则的作用

---

## 五、兼容性注意事项

### 5.1 OpenClash smart 功能支持

不同版本的 OpenClash 对 smart 功能的支持不同：

| 版本 | smart 支持 |
|------|-----------|
| 官方原版 OpenClash（Dev 分支） | ✅ 完整支持 smart-routing / smart-dns |
| 精简版/旧版本 OpenClash | ❌ 可能不支持 |

**本配置不依赖 smart 功能**：
- 使用标准的 `type: select` / `type: url-test` 策略组
- DNS 配置也是标准写法，不依赖 smart-dns
- 不管 OpenClash 支不支持 smart，配置都能正常使用
