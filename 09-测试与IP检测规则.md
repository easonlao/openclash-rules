# 测试与 IP 检测规则

## 零、项目重写原则

1. **不依赖外部仓库**：所有规则在本项目内自行维护创建，不依赖第三方规则仓库。
2. **按组审查确认**：每一组规则先确认设计，再进入完整配置。
3. **最小可用原则**：只保留必要的规则，不盲目复制大量个人偏好列表。

---

## 一、策略组设计

### 1.1 测试与 IP 检测统一策略组

所有测试与 IP 检测相关服务统一走 `测试` 策略组：

```yaml
proxy-groups:
  - name: 测试
    type: select
    proxies:
      - 通用选择    # 首选：默认走通用选择，统一节点
      # 其他节点...
```

**推荐默认选「通用选择」**：统一节点选择，切换「通用选择」就切换了所有测试服务的出口。

## 四、注意事项

### 4.1 测试时的节点切换

测试 IP 或 DNS 泄漏时：
- 可以手动切换 `测试` 策略组的节点，验证不同地区的节点效果
- 测试完成后记得切回默认的节点选择

### 4.2 DNS 泄漏测试特别注意

如果 DNS 泄漏测试发现泄漏：
1. 检查 Clash DNS 配置是否正确开启了 `enhanced-mode: fake-ip` 或 `redir-host`
2. 确认 `nameserver` 和 `fallback` 配置正确
3. 检查系统 DNS 是否被强制修改

### 4.3 速度测试建议

- 测试时建议关闭其他占用带宽的程序
- 连续测试 2-3 次取平均值，结果更准确
- speedtest.net 建议选择距离节点物理位置近的服务器

### 4.4 调试模式

如果遇到规则匹配问题：
1. 可以开启 Clash 的 `log-level: debug` 查看详细的匹配日志
2. 使用 Clash Dashboard 查看实时连接和规则匹配情况
3. 临时添加测试规则时，建议放在规则列表的最前面，确保优先匹配

## 五、已确认 ✅

本分组已确认（2026-06-17）：

| # | 设计点 |
|---|---------|
| 1 | 所有测试与 IP 检测相关服务统一走 `测试` 策略组 |
| 2 | `测试` 策略组默认推荐「通用选择」，统一节点选择 |
| 3 | 规则覆盖：IP 检测、速度测试、DNS 泄漏测试、Clash 相关工具 |
| 4 | 测试时可以手动切换节点验证效果，测试完成后记得切回默认 |
| 5 | 规则位置：在金融规则之后、国内直连之前 |
| 6 | **全局原则**：除了 AI 服务单独策略组，其他所有分组默认都走「通用选择」，切换通用就切换了所有服务的出口 |
